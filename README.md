# SwiftDeploy

A declarative deployment CLI that generates your entire stack from a single `manifest.yaml`, enforces policy gates via OPA, and provides live observability.

## Project Structure

```
project/
├── swiftdeploy                      ← CLI executable
├── manifest.yaml                    ← ONLY file you edit manually
├── app/
│   ├── main.py                      ← Python HTTP API service
│   ├── Dockerfile                   ← Single-stage Alpine build
│   └── requirements.txt
├── templates/
│   ├── nginx.conf.tmpl              ← Nginx config template
│   └── docker-compose.yml.tmpl      ← Compose template (includes OPA)
├── policies/
│   ├── infrastructure.rego          ← Disk/CPU/memory policy
│   ├── canary.rego                  ← Error rate/latency policy
│   └── data.json                    ← Policy threshold values
├── nginx.conf                       ← GENERATED by init
├── docker-compose.yml               ← GENERATED by init
├── history.jsonl                    ← GENERATED by status/deploy
└── audit_report.md                  ← GENERATED by audit
```

## Prerequisites

- Docker + Docker Compose v2
- Python 3.8+ with PyYAML: `pip install pyyaml`

## Setup

```bash
# 1. Clone the repo
git clone https://github.com/AnitaAliCloud/hng4-devops
cd hng4-devops

# 2. Make CLI executable
chmod +x swiftdeploy

# 3. Build the Docker image
docker build -t anitaali-swiftdeploy:latest ./app
```

## Subcommand Walkthrough

### `init` — Generate config files
```bash
./swiftdeploy init
```
Reads `manifest.yaml` → generates `nginx.conf` and `docker-compose.yml` from templates.

### `validate` — Pre-flight checks
```bash
./swiftdeploy validate
```
Runs 5 checks before touching Docker:
1. `manifest.yaml` exists and is valid YAML
2. All required fields are present and non-empty
3. Docker image exists locally
4. Nginx port is not already bound on host
5. `nginx.conf` is syntactically valid

### `deploy` — Start the stack
```bash
./swiftdeploy deploy
```
Runs `init` → `validate` → starts OPA → OPA infrastructure policy check → `docker compose up` → polls `/healthz` until healthy (60s timeout).

### `promote` — Switch modes
```bash
./swiftdeploy promote canary   # Switch to canary mode
./swiftdeploy promote stable   # Switch back to stable
```
- Scrapes `/metrics` and runs OPA canary safety check before promoting
- Updates `mode` in `manifest.yaml` in-place
- Regenerates `docker-compose.yml` with new `MODE` env var
- Restarts **only** the app container (Nginx and OPA keep running)
- Confirms new mode is active via `/healthz`

### `status` — Live dashboard
```bash
./swiftdeploy status
```
Live-refreshing terminal dashboard showing:
- Real-time req/s, P99 latency, error rate
- Policy compliance (infrastructure + canary safety)
- Appends every scrape to `history.jsonl`

### `audit` — Generate audit report
```bash
./swiftdeploy audit
```
Parses `history.jsonl` and generates `audit_report.md` with:
- Timeline of deploys, promotes, and teardowns
- Policy violations section

### `teardown` — Shut everything down
```bash
./swiftdeploy teardown           # Stop containers + remove network/volumes
./swiftdeploy teardown --clean   # Also delete generated files
```

## API Endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/` | GET | Welcome message with mode, version, timestamp |
| `/healthz` | GET | Liveness check, returns uptime in seconds |
| `/metrics` | GET | Prometheus text format metrics |
| `/chaos` | POST | Inject chaos (canary mode only) |

## Chaos Examples

```bash
# Slow mode
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 3}'

# Error mode
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'

# Recover
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

## OPA Policy Gates

### Infrastructure Policy (pre-deploy)
Blocks deployment if:
- Disk free < 10GB
- CPU load > 16.0
- Memory free < 10%

### Canary Safety Policy (pre-promote)
Blocks promotion if:
- Error rate > 1%
- P99 latency > 500ms

Threshold values are configured in `policies/data.json`.

## Nginx Access Log Format

```
$time_iso8601 | $status | ${request_time}s | $upstream_addr | $request
```

Example:
```
2026-05-06T14:18:43+00:00 | 200 | 0.001s | 172.19.0.2:3000 | GET /healthz HTTP/1.1
```
